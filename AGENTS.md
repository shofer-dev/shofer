# AGENTS.md

This file provides guidance to agents when working with code in this repository.

- Settings View Pattern: When working on `SettingsView`, inputs must bind to the local `cachedState`, NOT the live `useExtensionState()`. The `cachedState` acts as a buffer for user edits, isolating them from the `ContextProxy` source-of-truth until the user explicitly clicks "Save". Wiring inputs directly to the live state causes race conditions.

- File Change Tracking Pattern: Any new native tool that modifies workspace files MUST call both `task.fileContextTracker.captureOriginal(relPath, content)` before mutation and `task.fileContextTracker.trackFileContext(relPath, "shofer_edited")` after mutation. Without these, the file won't appear in the FileChangesPanel — no diff, revert, redo, or accept. Tools using `DiffViewProvider` get this automatically; tools doing direct filesystem writes (`insert_edit`, `sed`, `rename_symbol`, `file` for rm/mv, `generate_image`) must do it manually. See [`docs/file-change-tracking.md`](docs/file-change-tracking.md) for the full tracking coverage table.

- Task State Model: Task state is a two-axis structure `TaskState = { lifecycle: TaskLifecycle, rating?: CompletionRating }` defined in [`packages/types/src/history.ts`](packages/types/src/history.ts). Never conflate the two axes into a single string enum (the old `TaskExecutionState` mistake — values like `"completed_well"`). When adding a new lifecycle, update `TaskLifecycle`, `LIFECYCLE_VISUAL` in [`webview-ui/src/components/chat/TaskSelector.tsx`](webview-ui/src/components/chat/TaskSelector.tsx), and `sanitizeRestoredState` in [`src/services/task-manager/TaskManager.ts`](src/services/task-manager/TaskManager.ts) if the lifecycle is transient (cannot survive a restart).

- Single-Writer Persistence Rule: `TaskManager.setState(taskId, state)` is the **only** writer of `taskState` to both in-memory `ManagedTask.state` and persisted `HistoryItem.taskState`. Do NOT write `taskState` directly via `provider.updateTaskHistory({ ..., taskState })` from tools, event handlers, or `ShoferProvider`. Pass an `initialState` to `provider.createTask()` for new tasks; route all subsequent transitions through `setState`. This keeps in-memory and persisted views provably consistent and gives one choke point for invariants and telemetry.

- Single UI Source-of-Truth Rule: UI components must resolve task state via the fixed fallback chain `runtime?.state ?? item.taskState ?? IDLE_TASK_STATE`. The runtime overlay (live `ManagedTask`) always wins; persisted state is only a fallback when no live instance exists. Use the `resolveStateVisual` helper in [`TaskSelector.tsx`](webview-ui/src/components/chat/TaskSelector.tsx) for any state-dependent rendering — never branch on lifecycle/rating directly in component code.

- Self-Contained Events Rule: Lifecycle events (e.g. `TaskCompleted`, `TaskAborted` in [`packages/types/src/events.ts`](packages/types/src/events.ts)) must carry the data needed to interpret them — `TaskCompleted` carries `{ rating, isSubtask }`, `TaskAborted` carries `{ reason }`. Consumers must not have to call back into the task instance or `ShoferProvider` to figure out what happened. When adding a new lifecycle event, include all decision-relevant fields in the payload type.

- Restore-Ordering Rule: Methods on `TaskManager` that depend on the managed-task map being authoritative (e.g. `registerBackgroundTask`) must call `assertRestored()` at the top to prevent order-of-initialization bugs. `restoreManagedTasks` sets the `restored` flag exactly once after rehydrating from history and running `sanitizeRestoredState` to downgrade transient lifecycles to `idle`.

- Uniform Rendering Rule: Visual indicators for state-like enums must go through a single rendering mechanism (e.g. all task-state icons render via `<span class="codicon …">` with the `LIFECYCLE_VISUAL` + `RATING_VISUAL` overlay tables). Do not add SVG/`<svg>` special cases for individual values. New variants must be added to the lookup tables, not branched on at the call site.

- No Backward Compatibility Unless Asked: Persisted-state schema changes do not need migrations or compatibility shims unless the user explicitly requests them. Prefer clean and simple design over compatibility code. Bump the **minor** version (`Y` in `X.Y.Z`) when persisted state shape changes.

- Ask Categorization Rule: `ShoferAsk` is partitioned across exactly four _state_ categorizers in [`packages/types/src/message.ts`](packages/types/src/message.ts) — `isIdleAsk`, `isInteractiveAsk`, `isResumableAsk`, `isAgentRunningAsk` — describing how the agent loop _behaves_ while the ask is outstanding. Every ask MUST belong to exactly one of them; the partition test in [`packages/types/src/__tests__/message.test.ts`](packages/types/src/__tests__/message.test.ts) enforces this. The auto-approval policy (`isAutoApprovableAsk`) is a _separate, orthogonal_ predicate (does the host short-circuit the ask without user input?) and must NEVER be conflated with state classification by collapsing them onto a shared membership list. Conflation is exactly what the old `nonBlockingAsks` set did and is what produced the `attempt_completion` regression where queued messages and typed feedback were silently dropped.

- Auto-Approval Fast-Path Rule: An ask that ends a turn (i.e. anything in `idleAsks` such as `completion_result`, `api_req_failed`, `mistake_limit_reached`, …) MUST NOT be in `autoApprovableAsks`. The auto-approval fast-path in [`Task.ask()`](src/core/task/Task.ts) returns synchronously without entering `pWaitFor`, bypassing both the queue-drain logic and the `messageResponse` handling. Auto-approving an idle ask therefore causes (a) any user message queued during the task to be left in the queue forever, and (b) typed feedback to land on an aborted task and be dropped by the no-ask guard in `handleWebviewAskResponse`. The cross-policy invariant `isAutoApprovableAsk(ask) ⇒ isAgentRunningAsk(ask)` is asserted by the same test file above.

- Webview Send-Path Rule: [`ChatView.handleSendMessage`](webview-ui/src/components/chat/ChatView.tsx) MUST NOT post a bare `askResponse: "messageResponse"` without a confirmed pending ask (`shoferAskRef.current` set). When the user types in an ongoing task with no ask currently awaiting (e.g. the brief window between asks, or after an auto-approved ask cleared `shoferAsk`), route through `queueMessage` so the next `Task.ask()` drains it through the existing well-tested path. Relying on the host-side `prependMessage` fallback inside `handleWebviewAskResponse` is a defensive backstop, not a contract — the webview should never depend on it.

- Ordering of `addToShoferMessages` vs `checkAutoApproval`: When `Task.ask()` posts a new ask, `checkAutoApproval` MUST run _before_ `addToShoferMessages` so the `autoApproved` flag is set on the initial message persisted/posted to the webview. Inverting the order causes the Accept/Reject buttons to flicker momentarily before a follow-up `messageUpdated` retracts them. See the three near-identical fast-path blocks at the top of [`Task.ask()`](src/core/task/Task.ts) — keep them in sync if you touch one.

- Typed Settings Rule: All extension settings and secrets MUST go through [`ContextProxy`](src/core/config/ContextProxy.ts), not `vscode.workspace.getConfiguration("shofer.*")` or `context.secrets.get/store` directly. Add new state keys to `globalSettingsSchema` in [`packages/types/src/global-settings.ts`](packages/types/src/global-settings.ts) and new secret keys to `GLOBAL_SECRET_KEYS` in the same file; access via `ContextProxy.getInstance(context).getValue(key)` for state and `getSecret(key)` for secrets. The proxy auto-routes by key and gives one cache + one event stream for the whole extension. Bypassing it (as the original helper-agent did) splits the source-of-truth and breaks settings-view round-tripping.

- Typed Command Rule: Commands MUST be registered through the typed `CommandId` plumbing — add the id to the `commandIds` const in [`packages/types/src/vscode.ts`](packages/types/src/vscode.ts) and wire the handler in [`src/activate/registerCommands.ts`](src/activate/registerCommands.ts). Do NOT call `vscode.commands.registerCommand("shofer.foo", …)` ad-hoc from `extension.ts` or service constructors. Use `getCommand(id)` from [`src/utils/commands.ts`](src/utils/commands.ts) anywhere a prefixed command name is needed so the prefix is centralized.

- Shared `ApiHandler` Rule: Any service that talks to an LLM MUST go through `buildApiHandler(providerSettings, { taskId })` in [`src/api/index.ts`](src/api/index.ts) and consume the resulting `ApiHandler` interface — never instantiate provider SDKs (`@anthropic-ai/sdk`, `openai`, `@google/genai`, `ollama`, …) directly and never write bespoke `fetch()` calls per provider. The shared handler gives streaming, abort handling, the full provider catalog, model info / pricing, telemetry, and retries for free. Service code should depend only on `ApiHandler` + `ApiStream` chunk types so a new provider becomes available everywhere by editing the provider switch in one place. The helper-agent split (see [`src/services/helper-agent/llm-client.ts`](src/services/helper-agent/llm-client.ts)) is the reference pattern.

- Cooperative Cancellation Rule: Long-running async work (LLM calls, queue draining, watcher loops) MUST be cancellable via an `AbortSignal` threaded through the call stack — producer creates the controller, intermediate layers forward the signal, and the bottom layer either (a) checks `signal.aborted` between iterations / stream chunks and throws an `AbortError`, or (b) registers an `abort` listener that rejects its in-flight promise. Do NOT use polled boolean flags or rely on garbage collection of dropped promises. The pattern is established by [`QuestionQueue`](src/services/helper-agent/question-queue.ts) → [`HelperAgentLlmClient.chat`](src/services/helper-agent/llm-client.ts).

- Service Module-Split Rule: A service file that grows past ~500–700 lines and accretes orthogonal concerns (persistence, queueing, transport, accounting, watching) MUST be split into focused single-responsibility modules with a thin orchestrator that owns composition + lifecycle and re-exports the public API. Each module should be importable and testable without `vscode` mocks where possible. The helper-agent layout — `manager.ts` (orchestrator) + `conversation-store.ts` + `question-queue.ts` + `context-window.ts` + `llm-client.ts` + `pricing.ts` under [`src/services/helper-agent/`](src/services/helper-agent/) — is the reference template.

- Versioned Snapshot Rule: Persisted JSON snapshots MUST include an integer `version` field. On load, mismatched versions discard the snapshot and return an empty value (NOT a migration), consistent with the "No Backward Compatibility Unless Asked" rule above. Cached file contents inside a snapshot MUST be hash-validated (SHA-256 of current on-disk bytes vs stored `contentHash`) and dropped on mismatch or `ENOENT`. See [`ConversationStore.load`](src/services/helper-agent/conversation-store.ts) for the canonical implementation.

- Output Channel Logging Rule: Long-form / debug logging from extension code MUST go through the shared output channel logger ([`src/utils/outputChannelLogger.ts`](src/utils/outputChannelLogger.ts)), obtained via `getOutputChannel()` in [`src/extension.ts`](src/extension.ts) and wrapped with `createOutputChannelLogger` (or `createDualLogger` when also surfacing to dev console). Do NOT use bare `console.log` in extension-host code — the user has no way to see it inside VS Code, and CI/log scrapers cannot attribute it. Webview-side code may use `console.*` (it's the only thing the devtools panel sees), but prefer the IPC-forwarded logger when the message would be useful to a non-developer.

- Telemetry Capture Rule: All telemetry MUST go through `TelemetryService.instance.captureXxx(...)` typed methods in [`packages/telemetry/src/TelemetryService.ts`](packages/telemetry/src/TelemetryService.ts). New event kinds MUST be added to `TelemetryEventName` in [`packages/types/src/telemetry.ts`](packages/types/src/telemetry.ts) and exposed as a typed `captureXxx` wrapper — do NOT pass raw event-name strings to `captureEvent`, and do NOT instantiate `posthog` / `posthog-node` directly. Telemetry is opt-in and gated behind both the `TELEMETRY_ENABLED` build-time env flag (in [`PostHogTelemetryClient`](packages/telemetry/src/PostHogTelemetryClient.ts) and [`webview-ui/src/utils/TelemetryClient.ts`](webview-ui/src/utils/TelemetryClient.ts)) and the user's `TelemetrySetting`. Both gates MUST hold; do not bypass either.

- Telemetry Toggle Ordering Rule: When the user changes `telemetrySetting`, fire `captureTelemetrySettingsChanged(prev, next)` BEFORE calling `updateTelemetryState(false)` on a toggle-OFF, and AFTER `updateTelemetryState(true)` on a toggle-ON. This guarantees the change event itself is captured under the still-enabled side of the transition. The invariant is enforced by [`src/core/webview/__tests__/telemetrySettingsTracking.spec.ts`](src/core/webview/__tests__/telemetrySettingsTracking.spec.ts).

- Test Layout Rule: Tests live in `__tests__/` folders adjacent to the source they cover. Vitest is configured with `globals: true` everywhere, so do NOT import `describe`/`it`/`expect`/`vi` from `vitest`. Naming convention: extension-host and CLI use `*.test.ts(x)` (Node env); the webview uses `*.spec.ts(x)` (jsdom env, with the shared [`webview-ui/vitest.setup.ts`](webview-ui/vitest.setup.ts) for DOM polyfills and the `vscode` mock at [`webview-ui/src/__mocks__/vscode.ts`](webview-ui/src/__mocks__/vscode.ts)). Test verbosity is centralized via [`src/utils/vitest-verbosity.ts`](src/utils/vitest-verbosity.ts) — silent-by-default with `--no-silent` to enable; never wire ad-hoc `console.log` filtering per package.

- Module Boundaries Rule: `@shofer/types` is the single source of truth for cross-boundary schemas (asks, lifecycle events, telemetry events, settings keys, command ids, provider settings). Anything serialized over the webview ↔ host postMessage bridge, persisted to `globalState`/`secrets`, or sent to a telemetry backend MUST have its shape declared as a Zod schema in `packages/types/src/` and consumed by inferred `z.infer<>` types — not as a hand-written interface duplicated per consumer. New cross-boundary types belong in `@shofer/types`, never in the consuming package.
