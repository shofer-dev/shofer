# AGENTS.md

This file provides guidance to agents when working with code in this repository.

- Settings View Pattern: When working on `SettingsView`, inputs must bind to the local `cachedState`, NOT the live `useExtensionState()`. The `cachedState` acts as a buffer for user edits, isolating them from the `ContextProxy` source-of-truth until the user explicitly clicks "Save". Wiring inputs directly to the live state causes race conditions. See [`docs/settings_overlay.md`](docs/settings_overlay.md) for the full settings storage and merge architecture.

- File Change Tracking Pattern: Any new native tool that modifies workspace files MUST call both `task.fileContextTracker.captureOriginal(relPath, content)` before mutation and `task.fileContextTracker.trackFileContext(relPath, "shofer_edited")` after mutation. Without these, the file won't appear in the FileChangesPanel — no diff, revert, redo, or accept. Tools using `DiffViewProvider` get this automatically; tools doing direct filesystem writes (`insert_edit`, `sed`, `rename_symbol`, `file` for rm/mv, `generate_image`) must do it manually. See [`docs/file-change-tracking.md`](docs/file-change-tracking.md) for the full tracking coverage table.

- Task State Model: Task state is a two-axis structure `TaskState = { lifecycle: TaskLifecycle, rating?: CompletionRating }` defined in [`packages/types/src/history.ts`](packages/types/src/history.ts). Never conflate the two axes into a single string enum (the old `TaskExecutionState` mistake — values like `"completed_well"`). When adding a new lifecycle, update `TaskLifecycle`, `LIFECYCLE_VISUAL` in [`webview-ui/src/components/chat/TaskSelector.tsx`](webview-ui/src/components/chat/TaskSelector.tsx), `sanitizeRestoredState` in [`src/services/task-manager/TaskManager.ts`](src/services/task-manager/TaskManager.ts) if the lifecycle is transient (cannot survive a restart), `isTerminalLifecycle()` in [`packages/types/src/history.ts`](packages/types/src/history.ts) if adding a new terminal lifecycle, and [`docs/task_states.md`](docs/task_states.md) to keep the documentation in sync. When adding a new rating value, update `CompletionRating`, `RATING_VISUAL` in [`webview-ui/src/components/chat/TaskSelector.tsx`](webview-ui/src/components/chat/TaskSelector.tsx), and `completionRatingSchema` in [`packages/types/src/history.ts`](packages/types/src/history.ts). See [`docs/task_states.md`](docs/task_states.md) for the full state-model reference.

- Single-Writer Persistence Rule: `TaskManager.setState(taskId, state)` is the **only** writer of `taskState` to both in-memory `ManagedTask.state` and persisted `HistoryItem.taskState`. Do NOT write `taskState` directly via `provider.updateTaskHistory({ ..., taskState })` from tools, event handlers, or `ShoferProvider`. Pass an `initialState` to `provider.createTask()` for new tasks; route all subsequent transitions through `setState`. This keeps in-memory and persisted views provably consistent and gives one choke point for invariants and telemetry.

- Single UI Source-of-Truth Rule: UI components must resolve task state via the fixed fallback chain `runtime?.state ?? item.taskState ?? IDLE_TASK_STATE`. The runtime overlay (live `ManagedTask`) always wins; persisted state is only a fallback when no live instance exists. Use the `resolveStateVisual` helper in [`TaskSelector.tsx`](webview-ui/src/components/chat/TaskSelector.tsx) for any state-dependent rendering — never branch on lifecycle/rating directly in component code.

- Preload-Before-Publish Rule: A `Task` instance rehydrated from a `HistoryItem` MUST have its `shoferMessages` and `apiConversationHistory` populated from disk BEFORE it is pushed onto `ShoferProvider.shoferStack` (i.e. before it becomes `getCurrentTask()`). Construct the `Task` with `startTask: false`, `await task.preloadShoferMessages()` (idempotent, sets `historyPreloaded`), THEN call `addShoferToStack(task)` / perform the in-place swap, THEN `task.startFromHistory()` to drive `resumeTaskFromHistory()` (which short-circuits the redundant load via the `historyPreloaded` flag). Awaiting a `messagesReady` promise in the SAME caller after publishing does NOT close the race — any concurrent `postStateToWebview()` from another caller (a background task's `addToShoferMessages`, an unrelated webview round-trip, the `chatButtonClicked` navigation in `showTaskWithId`) sees the new task as `getCurrentTask()` with `shoferMessages: []` and broadcasts an empty-messages snapshot to the webview, which `ChatView` renders as the home screen — the "task-switch home-screen flash" symptom. Symmetrically, `showTaskWithId` MUST `await this.postStateToWebview()` between `createTaskWithHistoryItem` and the `chatButtonClicked` dispatch so the webview navigates to the chat view with the populated messages already in its cache. See [`Task.preloadShoferMessages`](src/core/task/Task.ts) and [`ShoferProvider.createTaskWithHistoryItem`](src/core/webview/ShoferProvider.ts).

- Self-Contained Events Rule: Lifecycle events (e.g. `TaskCompleted`, `TaskAborted` in [`packages/types/src/events.ts`](packages/types/src/events.ts)) must carry the data needed to interpret them — `TaskCompleted` carries `{ rating, isSubtask }`, `TaskAborted` carries `{ reason }`. Consumers must not have to call back into the task instance or `ShoferProvider` to figure out what happened. When adding a new lifecycle event, include all decision-relevant fields in the payload type.

- Self-Declared Terminal State Rule: A tool that represents the agent's own terminal state (currently only `attempt_completion`) MUST NOT call `task.ask(...)` to gate its emission of the lifecycle event. The completion `result`, `rating`, and optional `feedback` are produced by the agent itself — there is no user approval, no follow-up feedback prompt, and no waiting on `yesButtonClicked`. Render the result via `task.say("completion_result", …)`, persist completion artefacts, then synchronously call the event-emitting helper (`emitTaskCompleted`) and set `task.abort = true`. Using an `isIdleAsk` here (e.g. the old `task.ask("completion_result", "", false)`) is a bug: the `statusMutationTimeout` in `Task.ask()` fires `TaskIdle` → `TaskManager.setState(idle)` and the never-arriving response prevents `emitTaskCompleted` from running, so the rating overlay never lands in the UI. See [`docs/task_states.md`](docs/task_states.md) and [`AttemptCompletionTool.execute`](src/core/tools/AttemptCompletionTool.ts).

- Terminal-State Queue-Drain Rule: Because a self-declared terminal-state tool no longer calls `task.ask(…)`, it loses the queue-drain side-effect that `Task.ask()` provides for free (see [`docs/message_queue.md`](docs/message_queue.md)). The tool MUST therefore explicitly check `task.messageQueueService.isEmpty()` BEFORE the persist + `emitTaskCompleted` + `task.abort = true` block. If the queue is non-empty, dequeue the head (FIFO), render it via `task.say("user_feedback", …)`, push it as a tool result via `pushToolResult(formatResponse.toolResult(\`<user_message>\\n…\\n</user_message>\`, images))`, and `return`— letting the task loop continue to the next LLM iteration. Skipping this drain leaves the user's queued messages stranded and forces them to click **Send Now** to recover, which is exactly the symptom that prompted this rule. The remaining messages stay in the queue and are drained by the next`Task.ask()`. The complementary case — a NEW message arriving via `queueMessage`AFTER the task already aborted — is handled in the host's`queueMessage` handler ([`webviewMessageHandler.ts`](src/core/webview/webviewMessageHandler.ts)): if `currentTask.abort === true`at queue-enqueue time, the handler auto-triggers`cancelAndProcessQueuedMessages()` so the loop restarts with the new input. Queueing semantics apply only while the task is doing work; a completed task is not doing work.

- Message Queue Documentation Sync Rule: When adding, renaming, or changing the signature of any public method on [`MessageQueueService`](src/core/message-queue/MessageQueueService.ts) (`addMessage`, `prependMessage`, `dequeueMessage`, `removeMessage`, `updateMessage`, `isEmpty`, `dispose`), or when adding a new `QueueEvents` event variant, the corresponding sections of [`docs/message_queue.md`](docs/message_queue.md) MUST be updated in lock-step — specifically (a) the diagram in §"Components and responsibilities", (b) the FIFO ordering section if the new method participates in queue semantics, (c) the Files section enumerating the methods, and (d) the Gaps section if a previously-documented gap is addressed. Adding a method that affects the webview (new event emission) also requires an update to the [`QueuedMessages`](webview-ui/src/components/chat/QueuedMessages.tsx) component and its IPC plumbing in [`getStateToPostToWebview`](src/core/webview/ShoferProvider.ts).
- Restore-Ordering Rule: Methods on `TaskManager` that depend on the managed-task map being authoritative (e.g. `registerBackgroundTask`) must call `assertRestored()` at the top to prevent order-of-initialization bugs. `restoreManagedTasks` sets the `restored` flag exactly once after rehydrating from history and running `sanitizeRestoredState` to downgrade transient lifecycles to `idle`.

- Subtask Question Routing Rule: When a background child task (spawned via `new_task` with `is_background=true`) calls `ask_followup_question`, the question MUST be routed to the **parent task**, NOT to the user. The parent surfaces the pending question through `check_task_status` and answers it via the dedicated `answer_subtask_question` tool, which delivers the answer back to the blocked child as its `ask` response. Do NOT escalate background-child questions to the user via the webview ask UI — the user has no context for the child's prompt and the parent (which spawned the child with a specific goal) is the only correct decision-maker. Foreground/synchronous subtasks retain the normal user-ask flow because the user is actively watching that single conversation. The `cancel_tasks` tool is the parent-side escape hatch for terminating children whose pending questions cannot be answered productively. All four control-plane subtasks tools — `new_task`, `attempt_completion` (`finishTask`), `cancel_tasks`, `answer_subtask_question` — share the single `alwaysAllowSubtasks` auto-approval toggle in [`checkAutoApproval`](src/core/auto-approval/index.ts); they MUST stay grouped together (not split across ad-hoc allow-lists) so the toggle's meaning matches its label.

- Waiting Lifecycle Rule: A parent task that is blocked inside `wait_for_task` (or any future blocking-on-children primitive) MUST transition to the `waiting` lifecycle for the duration of the block and back to `running` (or the appropriate state) on resume. `waiting` is distinct from `idle` (the parent is not awaiting user input — it has live work in flight elsewhere) and from `running` (the parent's own LLM/tool loop is not advancing). `waiting` is transient and MUST be downgraded to `idle` by `sanitizeRestoredState` in [`TaskManager`](src/services/task-manager/TaskManager.ts) on restart, since no in-process wait survives a reload. Per the Task State Model rule above, also add the new lifecycle to `TaskLifecycle` and `LIFECYCLE_VISUAL` so the UI renders a distinct indicator rather than collapsing onto an existing state.

- Uniform Rendering Rule: Visual indicators for state-like enums must go through a single rendering mechanism (e.g. all task-state icons render via `<span class="codicon …">` with the `LIFECYCLE_VISUAL` + `RATING_VISUAL` overlay tables). Do not add SVG/`<svg>` special cases for individual values. New variants must be added to the lookup tables, not branched on at the call site.

- No Backward Compatibility Unless Asked: Persisted-state schema changes do not need migrations or compatibility shims unless the user explicitly requests them. Prefer clean and simple design over compatibility code. Bump the **minor** version (`Y` in `X.Y.Z`) when persisted state shape changes.

- Ask Categorization Rule: `ShoferAsk` is partitioned across exactly four _state_ categorizers in [`packages/types/src/message.ts`](packages/types/src/message.ts) — `isIdleAsk`, `isInteractiveAsk`, `isResumableAsk`, `isAgentRunningAsk` — describing how the agent loop _behaves_ while the ask is outstanding. Every ask MUST belong to exactly one of them; the partition test in [`packages/types/src/__tests__/message.test.ts`](packages/types/src/__tests__/message.test.ts) enforces this. The auto-approval policy (`isAutoApprovableAsk`) is a _separate, orthogonal_ predicate (does the host short-circuit the ask without user input?) and must NEVER be conflated with state classification by collapsing them onto a shared membership list. Conflation is exactly what the old `nonBlockingAsks` set did and is what produced the `attempt_completion` regression where queued messages and typed feedback were silently dropped. See [`docs/auto_approval.md`](docs/auto_approval.md) for the full auto-approval decision flow.

- Auto-Approval Fast-Path Rule: An ask that ends a turn (i.e. anything in `idleAsks` such as `completion_result`, `api_req_failed`, `mistake_limit_reached`, …) MUST NOT be in `autoApprovableAsks`. The auto-approval fast-path in [`Task.ask()`](src/core/task/Task.ts) returns synchronously without entering `pWaitFor`, bypassing both the queue-drain logic and the `messageResponse` handling. Auto-approving an idle ask therefore causes (a) any user message queued during the task to be left in the queue forever (see [`docs/message_queue.md`](docs/message_queue.md)), and (b) typed feedback to land on an aborted task and be dropped by the no-ask guard in `handleWebviewAskResponse`. The cross-policy invariant `isAutoApprovableAsk(ask) ⇒ isAgentRunningAsk(ask)` is asserted by the same test file above.

- Auto-Approval Documentation Sync Rule: When adding a new tool to the unconditionally-approved list in [`checkAutoApproval`](src/core/auto-approval/index.ts), changing its tool-group assignment in [`TOOL_GROUPS`](packages/types/src/tool.ts), or modifying which `alwaysAllow*` toggle gates it, [`docs/auto_approval.md`](docs/auto_approval.md) MUST be updated in lock-step — specifically (a) the Toggles table if a new toggle or additional-option column entry is needed, (b) the Unconditionally Auto-Approved section (meta-operations, background-task status, lightweight read-only, async MCP status) if the tool bypasses all toggles, (c) the Conditional Auto-Approval section (the per-toggle tool lists) if the tool is toggle-gated, and (d) the Related Files table if description text for an existing file is stale. This review found `give_feedback`, `check_mcp_call_status`, and `wait_for_mcp_call` unconditionally approved but undocumented; `cancel_tasks` and `answer_subtask_question` gated by `alwaysAllowSubtasks` but not listed; and `run_slash_command` listed under `alwaysAllowReadOnly` when it isn't even in `TOOL_GROUPS.read`. All were corrected.

- Tool Name Form Dual-Representation Rule: The auto-approval system operates on two different tool-name conventions: `ShoferSayTool.tool` (camelCase, e.g., `"runSlashCommand"`, `"newTask"`) in the `checkAutoApproval` path, and `ToolName` (snake_case, e.g., `"run_slash_command"`, `"new_task"`) in [`TOOL_GROUPS`](packages/types/src/tool.ts). The mapping between them lives in [`SAY_TOOL_TO_NATIVE_NAME`](src/core/auto-approval/tools.ts). When adding or reclassifying a tool, verify ALL THREE locations: (1) `TOOL_GROUPS` — the group determines what `alwaysAllow*` toggle gates the tool; (2) `SAY_TOOL_TO_NATIVE_NAME` — must exist for `getToolGroupForSayTool()` to resolve the group at all (missing entry → falls through to external-tool prefix inference → likely `"uncategorized"`); (3) `checkAutoApproval` — the unconditional-approval blocks and toggle-gated blocks in `index.ts` use the camelCase SayTool name directly. A tool in `TOOL_GROUPS.read` whose `SAY_TOOL_TO_NATIVE_NAME` entry is missing will silently receive the wrong auto-approval behavior. The bug with `run_slash_command` in docs occurred because someone inferred it was gated by `alwaysAllowReadOnly` without verifying the `TOOL_GROUPS.read` membership.

- Webview Send-Path Rule: [`ChatView.handleSendMessage`](webview-ui/src/components/chat/ChatView.tsx) MUST NOT post a bare `askResponse: "messageResponse"` without a confirmed pending ask (`shoferAskRef.current` set). When the user types in an ongoing task with no ask currently awaiting (e.g. the brief window between asks, or after an auto-approved ask cleared `shoferAsk`), route through `queueMessage` so the next `Task.ask()` drains it through the existing well-tested path. Relying on the host-side `prependMessage` fallback inside `handleWebviewAskResponse` is a defensive backstop, not a contract — the webview should never depend on it. See [`docs/message_queue.md`](docs/message_queue.md) for the queueing/Send-Now/per-task-drafts design.

- ChatView Draft-Snapshot Audit Rule: When modifying `ChatView`'s input state (`inputValue`, `selectedImages`, `droppedContextFiles`) or their derivative refs (`inputValueRef`), or when altering the control flow of `handleChatReset` or the task-id `useEffect`, the per-Task draft snapshot/restore flow MUST be re-audited. The invariant is: when a task switch occurs (task-id change), the outgoing task's current input state is snapshotted into `taskDraftsRef` BEFORE the incoming task's draft is restored. A change that clears `inputValue` before the `useEffect` fires, or that mutates `taskDraftsRef` in a way that causes the `useEffect` to snapshot stale state, breaks this invariant. See [`docs/message_queue.md`](docs/message_queue.md) §"Per-Task input drafts" for the full flow.
- Ordering of `addToShoferMessages` vs `checkAutoApproval`: When `Task.ask()` posts a new ask, `checkAutoApproval` MUST run _before_ `addToShoferMessages` so the `autoApproved` flag is set on the initial message persisted/posted to the webview. Inverting the order causes the Accept/Reject buttons to flicker momentarily before a follow-up `messageUpdated` retracts them. See the three near-identical fast-path blocks at the top of [`Task.ask()`](src/core/task/Task.ts) — keep them in sync if you touch one.

- Typed Settings Rule: All extension settings and secrets MUST go through [`ContextProxy`](src/core/config/ContextProxy.ts), not `vscode.workspace.getConfiguration("shofer.*")` or `context.secrets.get/store` directly. Add new state keys to `globalSettingsSchema` in [`packages/types/src/global-settings.ts`](packages/types/src/global-settings.ts) and new secret keys to `GLOBAL_SECRET_KEYS` in the same file; access via `ContextProxy.getInstance(context).getValue(key)` for state and `getSecret(key)` for secrets. The proxy auto-routes by key and gives one cache + one event stream for the whole extension. Bypassing it (as the original assistant-agent did) splits the source-of-truth and breaks settings-view round-tripping. See [`docs/settings_overlay.md`](docs/settings_overlay.md) for the three storage categories and merge semantics.

- Typed Command Rule: Commands MUST be registered through the typed `CommandId` plumbing — add the id to the `commandIds` const in [`packages/types/src/vscode.ts`](packages/types/src/vscode.ts) and wire the handler in [`src/activate/registerCommands.ts`](src/activate/registerCommands.ts). Do NOT call `vscode.commands.registerCommand("shofer.foo", …)` ad-hoc from `extension.ts` or service constructors. Use `getCommand(id)` from [`src/utils/commands.ts`](src/utils/commands.ts) anywhere a prefixed command name is needed so the prefix is centralized.

- Shared `ApiHandler` Rule: Any service that talks to an LLM MUST go through `buildApiHandler(providerSettings, { taskId })` in [`src/api/index.ts`](src/api/index.ts) and consume the resulting `ApiHandler` interface — never instantiate provider SDKs (`@anthropic-ai/sdk`, `openai`, `@google/genai`, `ollama`, …) directly and never write bespoke `fetch()` calls per provider. The shared handler gives streaming, abort handling, the full provider catalog, model info / pricing (see [`docs/cost-calculation-and-limits.md`](docs/cost-calculation-and-limits.md)), telemetry, and retries for free. Service code should depend only on `ApiHandler` + `ApiStream` chunk types so a new provider becomes available everywhere by editing the provider switch in one place. The assistant-agent split (see [`src/services/assistant-agent/llm-client.ts`](src/services/assistant-agent/llm-client.ts) and [`docs/assistant_agent.md`](docs/assistant_agent.md)) is the reference pattern.

- Dual Cancellation-Path Rule: `Task` has two distinct cancellation paths that MUST NOT be confused: (a) `abortTask()` — destructive tear-down called by user Stop, budget limit "kill", or stream failure. Sets `this.abort = true` then aborts `_taskAbortController`, then disposes the task. (b) `cancelAndProcessQueuedMessages()` — soft-cancel for Send Now. Sets `_softCancelForQueuedMessage = true`, aborts the old `_taskAbortController`, waits for the loop to exit, then **replaces** the controller with a fresh one and restarts the loop with the dequeued message. Code that initiates a new cancellation path MUST decide which variant it is and follow the matching protocol. Adding a third path that aborts `_taskAbortController` without setting `_softCancelForQueuedMessage = true` will cause the stream catch block to call `abortTask()` → `dispose()`, destroying the task when it should be restarted. See [`Task.abortTask`](src/core/task/Task.ts) and [`Task.cancelAndProcessQueuedMessages`](src/core/task/Task.ts).

- Abort-Ordering Invariant: In `abortTask()`, `this.abort = true` MUST be set BEFORE `_taskAbortController.abort()` so synchronous observers see the boolean flag first. Code that checks `task.abort` to decide whether to call `abortTask()` (see the stream catch block at [`Task.ts`](src/core/task/Task.ts) line ~4195) depends on this ordering. Reversing it produces a window where the signal is aborted but `this.abort` is still `false`, causing the catch block to skip `abortTask()` and leave the task in a zombie state.

- MCP Call-Site Indirection Rule: When code examples reference where `mcpHub.callTool()` or `mcpHub.readResource()` is invoked with `task.abortSignal`, the canonical call site is the shared helper [`runMcpToolCall`](src/core/tools/mcp/use-mcp-shared.ts:172), NOT the tool class files (`UseMcpToolTool.ts`, `accessMcpResourceTool.ts`) that delegate to it. The shared helper is where the signal is actually threaded into `McpHub.callTool(…, signal ?? task.abortSignal)`. Tool files that add their own MCP call sites MUST follow the same pattern: accept `signal?: AbortSignal` and fall back to `task.abortSignal`.

- Cooperative Cancellation Rule: Long-running async work (LLM calls, queue draining, watcher loops) MUST be cancellable via an `AbortSignal` threaded through the call stack — producer creates the controller, intermediate layers forward the signal, and the bottom layer either (a) checks `signal.aborted` between iterations / stream chunks and throws an `AbortError`, or (b) registers an `abort` listener that rejects its in-flight promise. Do NOT use polled boolean flags or rely on garbage collection of dropped promises. The pattern is established by [`QuestionQueue`](src/services/assistant-agent/question-queue.ts) → [`AssistantAgentLlmClient.chat`](src/services/assistant-agent/llm-client.ts). See [`docs/cancellation.md`](docs/cancellation.md) for the end-to-end Stop-button propagation flow.

- Service Module-Split Rule: A service file that grows past ~500–700 lines and accretes orthogonal concerns (persistence, queueing, transport, accounting, watching) MUST be split into focused single-responsibility modules with a thin orchestrator that owns composition + lifecycle and re-exports the public API. Each module should be importable and testable without `vscode` mocks where possible. The assistant-agent layout — `manager.ts` (orchestrator) + `conversation-store.ts` + `question-queue.ts` + `context-window.ts` + `llm-client.ts` + `pricing.ts` under [`src/services/assistant-agent/`](src/services/assistant-agent/) — is the reference template; see [`docs/assistant_agent.md`](docs/assistant_agent.md) for the full design rationale.

- Versioned Snapshot Rule: Persisted JSON snapshots MUST include an integer `version` field. On load, mismatched versions discard the snapshot and return an empty value (NOT a migration), consistent with the "No Backward Compatibility Unless Asked" rule above. Cached file contents inside a snapshot MUST be hash-validated (SHA-256 of current on-disk bytes vs stored `contentHash`) and dropped on mismatch or `ENOENT`. See [`ConversationStore.load`](src/services/assistant-agent/conversation-store.ts) for the canonical implementation.

- Output Channel Logging Rule: Long-form / debug logging from extension code MUST go through the shared output channel logger ([`src/utils/outputChannelLogger.ts`](src/utils/outputChannelLogger.ts)), obtained via `getOutputChannel()` in [`src/extension.ts`](src/extension.ts) and wrapped with `createOutputChannelLogger` (or `createDualLogger` when also surfacing to dev console). Do NOT use bare `console.log` in extension-host code — the user has no way to see it inside VS Code, and CI/log scrapers cannot attribute it. Webview-side code MUST NOT use `console.*` — route diagnostic messages back to the extension host via `vscode.postMessage` so they appear in the shared output channel.

- Telemetry Capture Rule: All telemetry MUST go through `TelemetryService.instance.captureXxx(...)` typed methods in [`packages/telemetry/src/TelemetryService.ts`](packages/telemetry/src/TelemetryService.ts). New event kinds MUST be added to `TelemetryEventName` in [`packages/types/src/telemetry.ts`](packages/types/src/telemetry.ts) and exposed as a typed `captureXxx` wrapper — do NOT pass raw event-name strings to `captureEvent`, and do NOT instantiate `posthog` / `posthog-node` directly. Telemetry is opt-in and gated behind both the `TELEMETRY_ENABLED` build-time env flag (in [`PostHogTelemetryClient`](packages/telemetry/src/PostHogTelemetryClient.ts) and [`webview-ui/src/utils/TelemetryClient.ts`](webview-ui/src/utils/TelemetryClient.ts)) and the user's `TelemetrySetting`. Both gates MUST hold; do not bypass either. See [`docs/telemetry.md`](docs/telemetry.md) for the full architecture.

- Telemetry Toggle Ordering Rule: When the user changes `telemetrySetting`, fire `captureTelemetrySettingsChanged(prev, next)` BEFORE calling `updateTelemetryState(false)` on a toggle-OFF, and AFTER `updateTelemetryState(true)` on a toggle-ON. This guarantees the change event itself is captured under the still-enabled side of the transition. The invariant is enforced by [`src/core/webview/__tests__/telemetrySettingsTracking.spec.ts`](src/core/webview/__tests__/telemetrySettingsTracking.spec.ts).

- Test Layout Rule: Tests live in `__tests__/` folders adjacent to the source they cover. Vitest is configured with `globals: true` everywhere, so do NOT import `describe`/`it`/`expect`/`vi` from `vitest`. Naming convention: extension-host and CLI use `*.test.ts(x)` (Node env); the webview uses `*.spec.ts(x)` (jsdom env, with the shared [`webview-ui/vitest.setup.ts`](webview-ui/vitest.setup.ts) for DOM polyfills and the `vscode` mock at [`webview-ui/src/__mocks__/vscode.ts`](webview-ui/src/__mocks__/vscode.ts)). Test verbosity is centralized via [`src/utils/vitest-verbosity.ts`](src/utils/vitest-verbosity.ts) — silent-by-default with `--no-silent` to enable; never wire ad-hoc `console.log` filtering per package.

- Module Boundaries Rule: `@shofer/types` is the single source of truth for cross-boundary schemas (asks, lifecycle events, telemetry events, settings keys, command ids, provider settings). Anything serialized over the webview ↔ host postMessage bridge, persisted to `globalState`/`secrets`, or sent to a telemetry backend MUST have its shape declared as a Zod schema in `packages/types/src/` and consumed by inferred `z.infer<>` types — not as a hand-written interface duplicated per consumer. New cross-boundary types belong in `@shofer/types`, never in the consuming package.

- Shared Module Isolation Rule: Anything under [`src/shared/`](src/shared/) is imported by BOTH the extension host AND the `webview-ui/` browser bundle (via the `@shofer/shared/*` path alias — see e.g. [`ExtensionStateContext.tsx`](webview-ui/src/context/ExtensionStateContext.tsx) and [`ExperimentalSettings.tsx`](webview-ui/src/components/settings/ExperimentalSettings.tsx) importing `@shofer/shared/experiments`). These modules MUST NOT statically `import` host-only APIs — most importantly `vscode`, but also Node built-ins (`fs`, `path`, `child_process`, …) and host services (`ContextProxy`, `TaskManager`, output channels, …). The webview bundler/runtime cannot resolve `vscode` and the module load throws silently, killing the React tree at startup — symptom: `resolveWebviewView` succeeds, HTML is assigned, but `WelcomeView` / `ChatView` / `SettingsView` never mount and the extension host logs look perfectly normal. If a helper needs `vscode`, put it in `src/activate/` (or another host-only directory) and import it from there — see the reference [`syncExperimentContextKeys`](src/activate/experimentContextKeys.ts) which was extracted out of `src/shared/experiments.ts` for exactly this reason.

- Native Tool Implementation Rule: Adding a native tool is a coordinated multi-file change spanning schema, types, handler, router, parser, and (when the tool renders UI) `ShoferSayTool` + `ChatRow` + i18n. The authoritative, up-to-date checklist is [`docs/adding-new-tools.md`](docs/adding-new-tools.md) — follow it rather than copying examples from existing tools (which may predate current conventions). The non-negotiable invariants are: handlers MUST extend `BaseTool<"name">` in [`src/core/tools/<Name>Tool.ts`](src/core/tools/); approval MUST go through `BaseTool.askToolApproval()` so even auto-approved invocations render in the chat UI; the tool's `ToolGroup` assignment in `TOOL_GROUPS` ([`packages/types/src/tool.ts`](packages/types/src/tool.ts)) is the single source of truth driving mode filtering AND auto-approval; tools MUST NOT branch on `mode` inside `execute()` — mode filtering happens upstream in `filterNativeToolsForMode`. External LM tools and MCP tools follow different paths; see [`docs/tool-categories.md`](docs/tool-categories.md).

- Advisory Parameter Defaults Rule: Native tool parameters described to the model as "soft" / "advisory" hints (e.g. `softResultLength`, `softTimeoutSec` on `new_task`) MUST be schema-optional (absent from JSON-schema `required` in [`src/core/prompts/tools/native-tools/`](src/core/prompts/tools/native-tools/)) AND have host-side defaults applied silently in the tool's `execute()` — never error on missing/invalid values via `sayAndCreateMissingParamError`. The two halves MUST stay coherent: the prompt cannot tell the model the field is advisory while the handler treats it as mandatory. Erroring on a parameter the prompt describes as optional causes a retry/churn cycle because the model has been told it's safe to omit it, then keeps re-issuing the call after the host rejects it. If the host genuinely needs the value, remove the "advisory" / "soft" / "optional" framing from the prompt and add it to `required`. See [`NewTaskTool.execute`](src/core/tools/NewTaskTool.ts) for the canonical pattern (default + clamp-to-cap for numeric soft hints).

- Native Tool Parser Cases Rule: A new native tool MUST add cases in **both** switches inside [`NativeToolCallParser`](src/core/assistant-message/NativeToolCallParser.ts) — `createPartialToolUse()` (streaming/partial args) AND `parseToolCall()` (complete args) — to construct typed `nativeArgs`. Adding the tool to `toolNames`, `TOOL_GROUPS`, the schema, the handler, and the router in [`presentAssistantMessage.ts`](src/core/assistant-message/presentAssistantMessage.ts) is **not enough**: if the parser has no case, `nativeArgs` is `undefined`, the dispatcher's `if (isKnownTool && !block.nativeArgs && !customTool)` guard rejects the call with `"Invalid tool call for '<name>': missing nativeArgs"`, and the tool's `execute()` is never reached. The failure surfaces in chat as a generic "Provider Error / API Request Failed" with no logs from the tool itself — symptom: one cost-ledger entry followed by an immediate `cancelTask`, and zero handler-side log lines. The two switches MUST stay in lock-step; use the existing `coerceOptionalNumber` / `coerceOptionalBoolean` / `Array.isArray(...)` helpers for optional fields, and validate required strings (`typeof x === "string"`) before assigning. This is step 7 of [`docs/adding-new-tools.md`](docs/adding-new-tools.md) and is the most commonly skipped step.

- Native Tool Documentation Sync Rule: Whenever a native tool is added, renamed, has its parameters changed, or has its `ToolGroup` reassigned, [`docs/native_tools.md`](docs/native_tools.md) MUST be updated in lock-step — specifically (a) the relevant **category summary table** row (Tool / Origin / Group / Always Available / Status / Description), (b) the per-tool **detail subsection** with its parameter table, AND (c) the **Mode × Tool Availability Matrix** at the bottom. Symmetrically, the `toolDescription()` switch in [`presentAssistantMessage.ts`](src/core/assistant-message/presentAssistantMessage.ts) MUST mirror the tool's actual parameter shape (e.g. plural `task_ids` array vs stale singular `task_id`) so chat rows render the same fields the parser populates; a mismatch here is silent — the call still executes but the chat row shows the wrong / partial / missing arguments. Doc-only changes do NOT require a version bump; parameter-shape changes do (per `No Backward Compatibility Unless Asked` — bump minor `Y`).

- Documentation-Source Coherence Rule: When a documentation file references named entities that exist in the source code — VS Code command IDs, configuration keys, TypeScript enum/union values, function signatures, file paths — those references MUST match the actual registered/compiled values exactly. A doc that says `shofer.vscodeTools.getDefinitions` when the command is registered as `arkware.vscodeTools.getDefinitions` is incorrect. A doc that omits enum members from a listed union type is incorrect. A doc relative file path that does not resolve is incorrect. These mismatches cause developers to write broken `settings.json` entries and waste time debugging silent failures. This rule applies to `docs/*.md` and any other user-facing documentation; it extends the Native Tool Documentation Sync Rule to cover interface contracts, config schemas, and command IDs. See the review of [`docs/tool-registration-interface.md`](docs/tool-registration-interface.md) for an example audit.

- Private-Tool Discovery Error Visibility Rule: Private tool provider discovery in [`getPrivateLmToolMeta()`](src/core/task/build-tools.ts) MUST NOT silently discard provider-level failures. A provider whose `getDefinitionsCommand` throws (extension not installed, not activated, or crashed) is silently skipped with no output-channel log and no user-facing indication. The user has no way to diagnose why a provider configured in `shofer.privateToolProviders` is not producing tools. At minimum, log the failure to the output channel; preferably surface it as a warning row in the chat UI so the user sees the misconfiguration immediately. The [`presentAssistantMessage.ts`](src/core/assistant-message/presentAssistantMessage.ts) invocation path has the same problem — a failing `invokeToolCommand` error should surface visibly, not silently drop the tool call.

- Linked-Profile Re-Init Rule: Any service that resolves and **caches** values from a linked API Configuration profile (today: `AssistantAgentManager`, which snapshots `handler.getModel().info.contextWindow` into `_config.maxContextTokens` at `initialize()`) MUST be re-initialized whenever **any** profile mutation occurs — `saveApiConfiguration`, `upsertApiConfiguration`, `renameApiConfiguration`, `loadApiConfiguration`, `loadApiConfigurationById`, `deleteApiConfiguration` — not only when the service's own `xxx*` settings change. The single re-init helper [`reinitializeAssistantAgent`](src/core/webview/webviewMessageHandler.ts) at the top of `webviewMessageHandler.ts` is the canonical hook; both the `updateSettings` branch and every profile-mutator case call into it. Without this, toggling e.g. `anthropicBeta1MContext` (200 K → 1 M) or switching the linked profile to a different model leaves the service pinned to the value it cached at activation time, which is exactly the bug that produced the popover showing a stale 128 K window. Corollary for the producer side: resolution helpers (e.g. `_loadConfiguration`) MUST surface failures (handler build errors, missing `info.contextWindow`) by **throwing** so `initialize()` promotes the service to `Error` state, NOT by silently substituting a magic constant like `DEFAULT_MAX_CONTEXT_TOKENS` — that hides exactly the problems this rule is designed to expose. The resolved-source provenance MUST be captured on the config (e.g. `contextWindowSource: "override" | "model-info" | "unresolved"`) and surfaced to the user (popover sub-line) so cache-vs-source mismatches are debuggable without spelunking.

- POSIX Path Presentation Rule: Any path that crosses an output boundary (shown to the model, persisted to `ShoferMessage`/`HistoryItem`, written to a JSON snapshot, or compared via `arePathsEqual`) MUST be normalized via `String.prototype.toPosix()` from [`src/utils/path.ts`](src/utils/path.ts). Use `path.relative(cwd, p).toPosix()` for relative paths and `getReadablePath(cwd, relPath)` when rendering for the user or AI. Never hand-roll `\\` → `/` replacement, and never present a raw Windows path to the model — it breaks fenced code blocks and downstream parsers. Native fs/VS Code APIs still receive native paths; the conversion is a presentation/serialization concern.

- i18n String Rule: User-facing strings MUST come from locale files. In the webview, use `const { t } = useAppTranslation()` from [`webview-ui/src/i18n/TranslationContext.tsx`](webview-ui/src/i18n/TranslationContext.tsx) and reference `t("namespace.key")`; add the key first to [`webview-ui/src/i18n/locales/en/<namespace>.json`](webview-ui/src/i18n/locales/en/) (English is the source of truth — other locales sync from it). In extension-host code, `import { t } from "./i18n"` and add to `src/i18n/locales/<lang>/`. Hard-coded English literals in JSX or `vscode.window.showXxxMessage(...)` calls are a bug.

- Webview Message Routing Rule: All webview → host messages MUST be a typed variant of `WebviewMessage` (declared in `@shofer/types`) and dispatched from the central [`webviewMessageHandler`](src/core/webview/webviewMessageHandler.ts). Do NOT add `vscode.postMessage(...)` calls with ad-hoc shapes from React components, and do NOT branch on `message.type` outside the handler. When a logical group of cases grows large or pulls in a heavy dependency, extract it into a sibling `*MessageHandler.ts` module (cf. [`skillsMessageHandler.ts`](src/core/webview/skillsMessageHandler.ts), [`webview/worktree/`](src/core/webview/worktree/)) and call into it from the central switch. Symmetrically, host → webview messages MUST be a typed `ExtensionMessage` variant.

- Exhaustive Switch Rule: Switches over discriminated unions in `@shofer/types` (e.g. `WebviewMessage["type"]`, `ShoferAsk`, `ShoferSay`, `ToolName`, `TaskLifecycle`, `TelemetryEventName`) MUST end with a `default:` branch that assigns the discriminant to a `never`-typed local (or calls a shared `assertNever(x)` helper). This converts "forgot to handle the new variant" from a silent runtime fallthrough into a compile error the moment the union is extended. Do NOT use `if (msg.type === "a" || msg.type === "b") … else { /* assume third */ }` for unions of more than two variants.

- Schema-First Persistence Rule: Any new persisted shape (file on disk, `globalState` value, `secrets` entry, IPC payload) MUST be defined as a Zod schema first — either in `@shofer/types` for cross-boundary shapes or in a co-located `*.schema.ts` for service-local ones — and reads MUST go through `schema.safeParse(raw)` so corrupt, partial, or stale on-disk data fails closed (returns `undefined`/empty and is overwritten on next write) rather than throwing from a type assertion or propagating `any` into business logic. Pair with the Versioned Snapshot Rule for top-level container shapes.

- Lazy Service Singleton Rule: Services with non-trivial initialization (`CodeIndexManager`, `AssistantAgentManager`, `MarketplaceManager`, `TaskManager`, `TelemetryService`, …) MUST expose a `static getInstance(context, cwd?)` accessor and be obtained through it — never `new XxxManager(...)` ad-hoc from a tool, command handler, or webview message handler. Modules that need them inside hot paths but would otherwise create a circular import at extension-activation time MUST use dynamic `await import(...)` at the call site (see the `CodeIndexManager` and `AssistantAgentManager` imports inside [`buildNativeToolsArrayWithRestrictions`](src/core/task/build-tools.ts)). Top-level static imports of those managers from `build-tools.ts` will reintroduce the activation-order bug they were extracted to fix.

- Protected Files Rule: Sensitive workspace configuration files (anything in `.shofer/`, `.vscode/`, `AGENTS.md`, `*.code-workspace`) are governed by [`ShoferProtectedController.PROTECTED_PATTERNS`](src/core/protect/ShoferProtectedController.ts). New sensitive paths MUST be added there, and write-paths in tools MUST gate on `isWriteProtected(relPath)` rather than special-casing filenames in tool implementations. The `SHIELD_SYMBOL` is the single canonical UI marker for a protected target — do not invent per-tool icons. See [`docs/shofer_special_files.md`](docs/shofer_special_files.md) for the full catalog of recognized files.

- Mode-Filtered Tool Exposure Rule: A tool's availability in a given mode is controlled exclusively through (a) its `ToolGroup` assignment in [`packages/types/src/tool.ts`](packages/types/src/tool.ts) / [`packages/types/src/mode.ts`](packages/types/src/mode.ts) and (b) the upstream filters `filterNativeToolsForMode` / `filterMcpToolsForMode` / `filterPrivateToolsForMode` in [`src/core/task/build-tools.ts`](src/core/task/build-tools.ts). Tools MUST NOT inspect the current mode inside `execute()` to short-circuit — by the time `execute()` runs, the mode filter has already authorized the call, and adding a second gate creates a second source of truth that drifts. See [`docs/tool_access.md`](docs/tool_access.md) for the `groups` / `tools_allowed` / `tools_denied` decision rules.

- Streaming Path-Stabilization Rule: Tools that update chat UI inside `handlePartial()` based on a `path` (or any string parameter sourced from partial-json streaming) MUST gate the update behind `hasPathStabilized(block.params.path)` from `BaseTool`, and MUST call `resetPartialState()` at the end of `execute()` on **both** success and error paths. Skipping the stabilization check causes truncated paths to flicker in the chat as chunk boundaries fall mid-value; skipping the reset causes the next invocation of the same tool to short-circuit on a stale path. See [`docs/tool-preparing-progress.md`](docs/tool-preparing-progress.md) for the broader tool-preparation progress indicator design.

- Radix `asChild` Trigger Rule: When wiring a Radix trigger primitive (`PopoverTrigger`, `DialogTrigger`, `TooltipTrigger`, `DropdownMenuTrigger`, …) with `asChild`, its **immediate** child MUST be the interactive element that owns the ref + DOM event handlers (typically `<Button>`), NOT a wrapper component like `StandardTooltip` that does not forward refs/handlers down to its inner element. Concretely: the working pattern is `<StandardTooltip><PopoverTrigger asChild><Button …/></PopoverTrigger></StandardTooltip>` — tooltip on the outside, trigger directly wrapping the button (see [`IndexingStatusBadge.tsx`](webview-ui/src/components/chat/IndexingStatusBadge.tsx)). The inverted layout `<PopoverTrigger asChild><StandardTooltip><Button …/></StandardTooltip></PopoverTrigger>` silently breaks click-to-open because Radix clones `StandardTooltip`, not the button. Likewise, a popover wrapper component MUST NOT bake `PopoverTrigger asChild` around `{children}` internally — leave trigger placement to the caller (cf. [`CodeIndexPopover`](webview-ui/src/components/chat/CodeIndexPopover.tsx) / [`AssistantAgentPopover`](webview-ui/src/components/chat/AssistantAgentPopover.tsx)).

- Tree-Sitter Language Registration Rule: When adding a new file extension to `CODEBASE_INDEX_FILE_EXTENSIONS` in [`packages/types/src/codebase-index.ts`](packages/types/src/codebase-index.ts), a matching `case` MUST be added in `loadRequiredLanguageParsers` in [`src/services/tree-sitter/languageParser.ts`](src/services/tree-sitter/languageParser.ts). Without it, `parseContent` throws `"Unsupported language: <ext>"` — silently dropping files from the RAG index (caught, emits `CODE_INDEX_ERROR` telemetry, returns `[]`) and crashing `list_code_definition_names` (uncaught). Additionally: extensions listed in `fallbackExtensions` in [`src/services/code-index/shared/supported-extensions.ts`](src/services/code-index/shared/supported-extensions.ts) early-return from `parseContent` before reaching the parser switch, making any switch case for them unreachable dead code — either remove the extension from `fallbackExtensions` to activate its parser, or remove the dead switch case. Finally: language queries in `src/services/tree-sitter/queries/*.ts` that use `#match?` / `#eq?` predicates MUST be validated against real-world code, not only test fixtures — a predicate matching exclusively test-fixture strings silently blocks all production files of that language from being indexed.

- Per-Provider Concurrency Lane Rule: External-API concurrency limiters (today: embedder providers in [`src/services/code-index/embedders/`](src/services/code-index/embedders/)) MUST be keyed by `(provider, endpoint)` at module scope and shared across every `IEmbedder` instance — not per-instance. The bug this prevents: each workspace/window creates its own `IEmbedder`, and an N-instance reindex storm bypasses any per-instance limit and trips the provider's rate limit (Ollama is the canonical victim with `OLLAMA_LANE = 1`; cloud providers default to `4`). The lane lives in a module-scoped `Map<string, PQueue>` (or equivalent) so multi-workspace and multi-task callers serialise through the same gate. Cooperative cancellation still applies — the lane MUST forward `AbortSignal` so a cancelled task doesn't hold the slot.

- Submodule-Aware Git Scanning Rule: Any code that walks files by consulting git (today: [`GitIgnoreFilter`](src/services/code-index/processors/ignore-filter.ts) and the git-history orchestrator/watcher under [`src/services/git-index/`](src/services/git-index/)) MUST descend into submodules declared in `.gitmodules`. Operationally: read `.gitmodules` once per scan root, recurse with the submodule path as the new `cwd`, and union the results — never assume the outer repo's index covers nested repos. Without this, files inside nested repos (e.g. `code-server/` under the workspace root) are invisible to the indexer and the git-history watcher, even though they're on disk.

- Indexer Policy Single-Source-of-Truth Rule: The indexer's static policy lists — currently `CODEBASE_INDEX_FILE_EXTENSIONS` and `CODEBASE_INDEX_IGNORED_DIRS` in [`packages/types/src/codebase-index.ts`](packages/types/src/codebase-index.ts) — are the sole definitions. Backend consumers ([`src/services/glob/constants.ts`](src/services/glob/constants.ts), [`src/services/tree-sitter/index.ts`](src/services/tree-sitter/index.ts)) re-export rather than redeclare; the webview ([`CodeIndexConfigForm.tsx`](webview-ui/src/components/settings/CodeIndexConfigForm.tsx) → Advanced Configuration) imports the same constants and renders them read-only so users can see the policy without grepping. New entries belong in `@shofer/types`; never inline a literal list at a call site.

- No Ad-Hoc VS Code Config Reads Rule: `vscode.workspace.getConfiguration("shofer.*")` and `context.globalState.get/update` MUST NOT appear in feature code. Only three kinds of call sites are allowed to bypass `ContextProxy`: (1) `ContextProxy` itself; (2) the registration boundary reading non-shofer settings declared in `ALLOWED_VSCODE_SETTINGS` (e.g. `terminal.integrated.inheritEnv` in [`webviewMessageHandler.ts`](src/core/webview/webviewMessageHandler.ts)); (3) extension-point discovery whose schema lives in `package.json` (e.g. `shofer.privateToolProviders` in [`build-tools.ts`](src/core/task/build-tools.ts) — see [`docs/tool-registration-interface.md`](docs/tool-registration-interface.md)). Everywhere else, go through `ContextProxy.getInstance(context).getValue(key)` so settings are typed, cached, and emit change events.

- GitIgnore Oracle Rule: The code-index scanner, file-watcher, and service-factory MUST accept `IIgnoreFilter` (from [`src/services/code-index/shared/git-ignore-filter.ts`](src/services/code-index/shared/git-ignore-filter.ts)) rather than the npm `Ignore` type directly at every call site. At service-creation time in `CodeIndexManager._recreateServices()`, prefer `GitIgnoreFilter.create(workspacePath)` — which runs `git ls-files -z --cached --others --exclude-standard` and therefore honours nested `.gitignore` files, `.git/info/exclude`, the global `core.excludesfile`, and negation patterns — over a flat root-only `ignore().add(content)` parse. Fall back to the flat parse only when `GitIgnoreFilter.create()` returns `null` (non-git workspace or git binary unavailable). A `**/.gitignore` `FileSystemWatcher` (managed via `_ensureGitIgnoreWatcher` / `_disposeGitIgnoreWatcher` in `CodeIndexManager`) refreshes the set on rule changes, debounced to 500 ms to survive branch switches. Do NOT reintroduce `import { Ignore } from "ignore"` at any call site in the code-index subsystem.

- Doc Code-Example Line-Number Rule: When a documentation file includes a code example annotated with a source line number (e.g., `// Task.ts line ~3150` or `lines 2344-2350`), that line number MUST be verified against the current source at the time of writing. Line numbers drift as files grow; an annotation that is off by 200+ lines misleads readers who try to follow along in the source. Prefer referencing stable symbolic names (function names, variable names) over line numbers when possible. This rule was discovered during the [`cost-calculation-and-limits.md`](docs/cost-calculation-and-limits.md) review where every code-example line number was off by 170–300 lines because `Task.ts` had grown from ~3,250 to 6,122 lines.

- Doc Code-Example Completeness Rule: When a documentation file includes a code example from source, it MUST include all non-trivial fields and guards present in the actual code. An `api_req_started` example that shows `JSON.stringify({ apiProtocol })` when the source writes `JSON.stringify({ apiProtocol, model: modelId, retryAttempt: currentItem.retryAttempt ?? 0 })` is incomplete and misleads readers about what data is actually persisted. If the example is intentionally simplified, mark it with a `(simplified)` annotation so the reader knows to consult the source for the full version. This rule was discovered during the [`cost-calculation-and-limits.md`](docs/cost-calculation-and-limits.md) review where the `consolidateTokenUsage.ts` example omitted `message.type === "say"` guards and `typeof` numeric checks present in the real code.

- Docs-Implementation Coherence Rule: When a design doc describes runtime behavior (timers, intervals, emission schedules, algorithms, data flow), every claim MUST be verifiable against the actual source code. Before publishing a doc, run the described code paths and confirm each behavioral claim matches the implementation. Variables assigned but never read, timer-based heartbeats that don't exist in code, or claims about emission schedules that don't match the code are doc-implementation mismatches that mislead readers. Use `grep_search` to verify named entities exist and `read_file` to confirm logic matches prose. See [`docs/tool-preparing-progress.md`](docs/tool-preparing-progress.md) for an example corrected after such verification.
- Circular-Import Lazy Logger Rule: Service files that cannot statically import `getOutputChannel()` from [`src/extension.ts`](src/extension.ts) without forming a require-cycle (because `extension.ts` transitively imports the service) MUST use a lazy dynamic import to write to the output channel: `void import("path/to/extension").then(({ getOutputChannel }) => { const ch = getOutputChannel(); if (!ch) return; ch.appendLine(\`...\`); }).catch(() => { /_ channel not yet wired; drop silently _/ })`. Do NOT substitute `console.log` as a workaround — that violates the Output Channel Logging Rule. See [`src/services/code-index/processors/file-watcher.ts`](src/services/code-index/processors/file-watcher.ts) for the reference `log()` helper implementation.

- Doc Anchor Coherence Rule: When a markdown documentation file has a Table of Contents with section anchor links (e.g., `[Some Section](#1-some-section)`), every anchor MUST match the section heading rendered text exactly. A ToC entry referencing `#6-drag--drop-system` when the heading is `## 6. Drag & Drop Workaround` produces a broken link. The fix: read the heading text and match it byte-for-byte, including spaces and punctuation — GitHub-flavored markdown slugifies `&` and other special characters into `-`. This review found 3 such mismatches in `docs/shofer_for_roocode_users.md`.

- Doc Entity Count Coherence Rule: When a documentation file states a numeric count of items (e.g., "Twelve native tools were implemented"), that count MUST match the actual number of items enumerated in the adjacent list or table, AND must be consistent with the latest source. A doc that says "Twelve" but lists 17 items is contradictory. Prefer omitting a count entirely; if you must state one, verify it against the enumerated list at the time of writing.

- Doc Enum Subset Rule: When a documentation file describes members of a source-code enum or union type (e.g., `TaskLifecycle`), it MUST either (a) list the complete set, or (b) explicitly qualify the list as a subset with language like "including" or "core states". A doc that says task states are "running, paused, completed, error" without disclosing that the full enum has 7 values is misleadingly incomplete. This review found the doc omitted `idle`, `waiting_input`, and `waiting`.

- Terminology Sync Rule: When a native tool is added, renamed, has its parameters changed, or has its `ToolGroup` reassigned, [`docs/terminology.md`](docs/terminology.md) MUST be updated in lock-step — specifically (a) the tool added to the appropriate category subsection in §9, and (b) the deprecated-names table updated if aliases change. Similarly, when a new `TaskLifecycle` value is added, update the `TaskLifecycle` row in §7 and the State Indicator Colors table in §12. When a new type is added to `@shofer/types`, add it to the appropriate type table in §7. Doc-only changes do NOT require a version bump.

- Type-Documentation Coherence Rule: When a documentation file includes a type-shape literal as a code example (e.g. `{ id: string, content: string, completed: boolean }`), that literal MUST match the actual Zod schema shape in source code. A doc that says `completed: boolean` when the schema has `status: "pending" | "in_progress" | "completed"` is incorrect and will cause developers to write broken code. This is a corollary of the Schema-First Persistence Rule — the schema is the single source of truth; docs must reflect it.

- Documentation File-Path Verification Rule: Every file path referenced in documentation (link or code reference) MUST resolve to an existing file in the repository. When writing or updating docs, verify every path with `list_files` or `find_files`. A broken path in a documentation file is a bug that wastes developer and agent time. This applies especially to `docs/terminology.md` which lists file paths for every UI component and backend service.

- Documentation Enum-Coherence Rule: Every enum value listed in documentation (e.g., TaskLifecycle members, ToolGroup names, CompletionRating values) MUST be the complete and exact set from the source. An enum documented with 6 members when the source has 7 is incorrect. Remove any phantom enum values that don't exist in the source, and add any missing ones. This rule applies to any documentation table or prose that enumerates union/enum members.

- Documentation-to-Code Verification Rule: When writing or updating documentation that references named code entities — file paths, function/method names, class names, interface names, enum members, type aliases, constants, line numbers — those references MUST be verified against the actual codebase at the time of writing. Use `grep_search`, `list_files`, or `read_file` to confirm existence and correct line numbers. Do not copy entity names from memory, from other docs, or from stale snapshots. A doc that references a non-existent file path, a phantom function name, or a wrong line number is incorrect and will cause developers and agents to waste time debugging silent mismatches. See the review of [`docs/telemetry.md`](docs/telemetry.md) for an example audit that uncovered ~60 line-number drifts, 6 missing enum members, and references to 2 non-existent source files.

- Documentation-Verifiable-Entities Rule: Any named entity referenced in documentation — a function, constant, config key, file path, or type — MUST exist in the codebase at the stated location. When writing docs that reference code, verify every function name (e.g., `resolveExternalLmToolGroup()` — never existed), every config key (e.g., `shofer.browserTools.toolGroups` — never existed), and every constant (e.g., `alwaysAvailable` on `ToolGroupConfig` — never existed) against the actual source. Phantom entities in docs mislead developers and agents.

- Tool-Group Documentation Sync Rule: When changing `TOOL_GROUPS`, `ALWAYS_AVAILABLE_TOOLS`, or `DEFAULT_MODES` in [`packages/types/src/tool.ts`](packages/types/src/tool.ts) or [`packages/types/src/mode.ts`](packages/types/src/mode.ts), [`docs/tool-categories.md`](docs/tool-categories.md) MUST be updated in lock-step — specifically (a) the `TOOL_GROUPS` code example, (b) the always-available tools list, AND (c) the Mode × Allowed Groups table. Skipping this produces docs that assert wrong group membership, wrong always-available tool names, and wrong mode permissions — the exact kinds of errors discovered during verification of this doc. This rule is the doc-side counterpart of the existing "Native Tool Documentation Sync Rule" (which covers `docs/native_tools.md`).

- Documentation Config-Namespace Rule: When writing docs that reference VS Code configuration namespaces (e.g., `arkware.vscodeTools.toolGroups`), verify the namespace against the extension's actual `package.json` `contributes.configuration.properties`. Do not assume a `shofer.*` prefix — the actual namespace is `arkware.*`. This applies to any config key shown in documentation code examples.

- Tool-Group Dual-Resolution Rule: External LM tools (registered by companion extensions like `arkware-vscode-tools` and `arkware-browser-tools`) have their groups resolved via two independent paths that MUST agree: (1) `filterPrivateToolsForMode` in [`filter-tools-for-mode.ts`](src/core/prompts/tools/filter-tools-for-mode.ts) reads each tool's group from the extension's `toolGroups` config for mode filtering; (2) `getToolGroupForSayTool` in [`auto-approval/tools.ts`](src/core/auto-approval/tools.ts) infers the group from the tool name prefix for auto-approval decisions. When adding a new external-tool prefix or changing an extension's config namespace, BOTH paths must be updated or the tool will be classified inconsistently across mode filtering and auto-approval.

- Documentation Interface Field-Ordering Rule: When a documentation file includes TypeScript interface or type definitions, the field **order** MUST match the source exactly. A field placed in the wrong position relative to surrounding fields (e.g., `_tokensEstimated` between `reasoning` and `retryAttempt` when the source has it after `wireRequest`) is a bug that will cause readers to misunderstand the data shape. This extends the Documentation-Source Coherence Rule — correctness includes ordering, not just existence.

- Documentation Example-Value Verification Rule: Inline comments in doc code examples that list possible values (e.g., `apiProtocol?: string // "anthropic", "openai", "openai-native"`) MUST be verified against the actual **runtime** value space, not merely copied from source comments. Source comments can themselves be wrong (e.g., the `export-json.ts:19` comment says `"openai-native"` but [`getApiProtocol()`](packages/types/src/provider-settings.ts:549) only ever returns `"anthropic" | "openai"`). Use `grep_search` to trace constant returns and enum members before documenting their values.

- Documentation UI-Icon Rule: When documenting UI buttons that use an icon library (lucide-react, codicon, etc.), reference the actual component or icon name (e.g., `DownloadIcon` from `lucide-react`), not an invented approximation (emoji, ASCII art). A Quick Reference table row that says `⬇ Download` when the real button renders a lucide `DownloadIcon` SVG is inaccurate and breaks for users trying to identify the UI element. Verify against the source component file.

- Doc-Code Sync Rule: Documentation files in `docs/` that reference source code entities MUST be verified against the live codebase. This means: (a) every file path link MUST resolve to an existing file; (b) every code example MUST be identical to the corresponding source (no simplifications, no elided parameters, no "abridged for clarity"); (c) every named entity (class, interface, enum, function, method, type field) MUST exist with the exact name and signature described; (d) every IPC message type and payload field MUST match the schema. A doc that claims to show a code snippet from `handlers.ts` but simplifies `path.relative(path.resolve(gitRootPath), path.resolve(cwd))` to `path.relative(gitRootPath, cwd)` is incorrect; (e) every quoted file-content literal (e.g., a code block showing the bytes inside a config file) MUST match the on-disk bytes exactly, including case — a doc quoting `gitdir: ../../.git/modules/Shofer` when the file contains `shofer` (lowercase) is incorrect. Apply the verification workflow used in the [`worktrees.md`](docs/worktrees.md) review: resolve every path, diff every code block against source, and enumerate every named entity. See the review of [`submodule-support.md`](docs/submodule-support.md) for an example audit that caught wrong case in a quoted file-content literal and an entire section describing a backend architecture that never existed in the code.

- Documentation Fabricated-Content Rule: Documentation MUST NOT include fabricated error messages, example outputs, or behavioral claims that do not exist in the source code. When documenting error handling, verify the exact error text in the source via `grep_search` — do not invent plausible-sounding wording. An error message block quoted in docs that does not exist in any source file is misinformation that misleads developers debugging issues. This rule covers both inline code-fenced "example output" blocks and prose claims about what the system says/does. See the review of [`docs/shofer_special_files.md`](docs/shofer_special_files.md) § "Fabricated `.shofer/shoferignore` error message" for the motivating case.

- Documentation Architecture-Claim Rule: When a documentation file makes claims about system architecture (e.g., "has two backends," "uses a fallback mode," "shows a limited-mode badge"), those claims MUST be verified against the actual source code. An architectural claim that contradicts the code is worse than a missing file path — it teaches developers an incorrect mental model of the system. Before publishing or updating docs that describe component relationships, read the relevant source files and confirm every architectural assertion. The review of [`submodule-support.md`](docs/submodule-support.md) caught a section claiming a two-backend (checkpoint + tracker) architecture for the file-changes panel when the code has always used a single working-directory backend.

- Dead-Code Discovery Rule: When a doc references a source file as the implementation of a documented feature, verify that the class/file is actually imported and used somewhere — not merely defined. A class with zero import references across the entire extension is dead code, and documenting it as an active implementation is misleading. Either integrate the dead code, remove it, or document it accurately as "defined but not yet wired."

- Singleton Re-export Rule: When a service class is designed for singleton use with no per-instance state beyond what its methods accept, co-locate a pre-constructed instance export alongside the class definition (e.g., `export const worktreeService = new WorktreeService()` in [`worktree-service.ts`](packages/core/src/worktree/worktree-service.ts)). This avoids `new` boilerplate at every call site and keeps the singleton contract visible at the definition site. Symmetrically, when a singleton instance is exported, the class itself SHOULD NOT be instantiated externally.

- Embedded Worktree Convention Rule: When creating or referencing git worktrees from within Shofer, all worktrees MUST live under `<workspace>/.shofer/worktrees/<name>/`. Callers that pass a path outside this prefix MUST have it normalized by prepending the convention prefix (see [`handleCreateWorktree`](src/core/webview/worktree/handlers.ts:172-182)). Handlers that accept a `cwdOverride` to query worktree state for non-workspace-cwd worktrees (e.g., [`handleGetWorktreeStatus`](src/core/webview/worktree/handlers.ts:298)) MUST thread this parameter through so parallel tasks in embedded worktrees get accurate git status regardless of the VS Code window's cwd. New handlers added to the worktree subsystem MUST follow the same pattern: accept an explicit cwd parameter rather than reading `provider.cwd` directly.

- Worktree Handler Naming Rule: All worktree webview handler functions exported from [`src/core/webview/worktree/handlers.ts`](src/core/webview/worktree/handlers.ts) MUST be named `handle<Verb><Noun>` (e.g., `handleListWorktrees`, `handleCreateWorktree`, `handleGetWorktreeStatus`), accepting `provider: ShoferProvider` as the first parameter. When a new IPC message type is added, the corresponding handler MUST be added to the same module and exported, rather than being inlined in the central switch of [`webviewMessageHandler.ts`](src/core/webview/webviewMessageHandler.ts).

- Export Schema Co-Evolution Rule: The JSON export subsystem ([`export-json.ts`](src/integrations/misc/export-json.ts)) reads per-call metadata from `api_req_started` ShoferMessages whose payload shape (`ShoferApiReqInfo`) is written in [`Task.ts`](src/core/task/Task.ts) and parsed as `UiApiReqStartedPayload` in export-json.ts. Adding a field to the write side (`ShoferApiReqInfo` → `api_req_started` JSON) without a matching key in `UiApiReqStartedPayload` AND a destination field on `JsonExportCall` silently drops that field from JSON exports. The three representations — `ShoferApiReqInfo` (write), `UiApiReqStartedPayload` (read/parse), `JsonExportCall` (export output) — MUST stay in lock-step. See [`docs/task-export.md`](docs/task_export.md) for the full export pipeline.

- Wire-Capture Anchor Rule: `snapshotWireRequest()` and `snapshotApiReqError()` in [`Task.ts`](src/core/task/Task.ts) use `findLastIndex` to locate the most recent `api_req_started` ShoferMessage for in-place mutation. The ordering invariant is: (1) emit `api_req_started` to `this.shoferMessages`, (2) optionally call `snapshotWireRequest()` / `snapshotApiReqError()`, (3) call `this.api.createMessage()`. Inserting new ShoferMessages between steps 1 and 2 silently redirects the wire request / error merge into the wrong `api_req_started` entry, producing corrupted exports. Any new message-emission code that runs between API-request setup and `createMessage()` MUST consider this anchor ordering.

- Export Resilient-Read Rule: Code that reads persisted task JSON files (`ui_messages.json`, `api_conversation_history.json`, `history_item.json`) MUST guard against missing files and JSON parse failures, following the canonical pattern in [`ShoferProvider.exportTaskWithIdJson`](src/core/webview/ShoferProvider.ts): (1) `fs.stat` existence check, (2) `try/catch` around `JSON.parse`, (3) fallback to an empty/zeroed default (`[]`, `{}`, `0`). Failing closed preserves partial export functionality; throwing on a corrupt file loses the entire export and frustrates the user. See also the Schema-First Persistence Rule — the same `safeParse` pattern applies to all persisted reads.
